# CocoaSkills RFC 0002 — Source URL Resolution

Status: accepted
Date: 2026-05-18
Author: Ivan Oparin
Target: v0.4.0

## 1. Goals

- Allow `Skillfile.json` to declare a clone URL for each skill so that `csk`
  can bootstrap missing source repositories.
- Make new project repositories self-contained: a fresh clone plus
  `csk install` should succeed without manual `git clone` of every skill.
- Preserve backward compatibility: existing `Skillfile.json` without `git`
  fields keeps working.
- Delegate authentication entirely to the user's existing git credentials.

## 2. Non-Goals

- Schema bump. `Skillfile.json` remains at `schema_version: 1`.
- Machine-wide source alias registry (`skill_sources` map). Deferred to a
  future RFC.
- Public skill registry / discovery service. Deferred to v1.0+.
- Shallow clones. Full clone is required for tag and revision resolution.
- Credential management. Delegated to git: SSH agent, credential helper, etc.
- Validation that an existing local clone's remote matches Skillfile `git`.
- Automatic remote URL rewrite when Skillfile `git` changes.
- Negative clone outcome cache. A failing URL is retried on the next install.

## 3. Motivation

After v0.3.0, `csk` handles per-project installation cleanly via `csk init` and
`csk install`. But the source skills referenced by `Skillfile.json` must
already exist under `skills_root` (typically `~/agents/skills/`). New users
hit this wall on first use:

```
$ git clone demo-app
$ cd demo-app
$ csk install
error: Skill repository not found for skill-review:
       /Users/x/agents/skills/skill-review
```

There is no signal from `Skillfile.json` telling `csk` where to get
`skill-review` from. The user must manually clone each declared skill before
`csk install` works. This makes `csk` a "manage-already-cloned-skills" tool
rather than a real dependency manager.

CocoaPods, npm, cargo, and bundler all bootstrap source code automatically on
first install. v0.4.0 brings `csk` in line with this convention.

## 4. Skillfile.json Schema Delta

Each skill declaration MAY include an optional `git` field with a clone URL:

```json
{
  "schema_version": 1,
  "project": {"alias": "demo-ios"},
  "skills": [
    {
      "name": "skill-review",
      "git": "git@gitlab.example.com:agents/skill-review.git",
      "revision": "5ad3663214883f944cbf15f7eb5b425e4bccfdec"
    },
    {
      "name": "skill-tracker",
      "git": "https://github.com/example/skill-tracker.git",
      "tag": "v1.0.0"
    },
    {
      "name": "skill-legacy",
      "revision": "abc123"
    }
  ]
}
```

`git` is optional. Entries without it keep current behavior.

Accepted URL forms: any URL `git clone` accepts — SSH (`git@host:path`,
`ssh://user@host/path`), HTTPS (`https://host/path`), file paths
(`/abs/path`, `file:///abs/path`). `csk` does not parse or validate the URL
itself; it forwards to git.

`source` field interaction: when `source` is set (renaming the local source
directory), the clone target is `skills_root/<source>/`. When `source` is
absent, clone target is `skills_root/<name>/`.

Local path URLs are not pre-validated by `csk`. If `git` points to a missing
local path, `git clone` fails and the failure is reported through the same
clone error path as SSH/HTTPS failures.

## 5. Resolution Algorithm

For each skill declaration during `install` or `upgrade`:

1. Compute `local_path = skills_root / (source or name)`.
2. If `local_path` exists and contains `.git/`, use it as the source.
   Unchanged from v0.3.x.
3. If `local_path` does not exist:
   - If `git` field is present, run `git clone <git> <local_path>` and
     proceed with normal install flow on the cloned repository.
   - If `git` field is absent, fail with the existing error
     `Skill repository not found for <name>: <local_path>`.
4. If `local_path` exists but is not a git repository, fail with
   `Local skill path exists but is not a git repository: <local_path>`.
   `csk` never wipes or modifies an existing non-git directory.

## 6. Clone Semantics

- Full clone, not shallow. `csk` needs all refs and history for tag and
  revision resolution.
- Default branch tracking. No `--branch` flag passed to `git clone` — `csk`
  resolves the ref via existing `git_ops.resolve_ref` after clone.
- Clone target is exactly `skills_root/<source or name>/`. Parent directory
  created if missing.
- Authentication is git's responsibility. `csk` neither prompts for nor reads
  credentials. If clone fails for auth reasons, the user sees git's native
  error wrapped in a clean `csk` message.
- Network failures during clone produce a clean per-project failure; other
  projects continue when running `csk install --all`.

## 7. Dry-Run Behavior

`csk install --dry-run` MUST NOT clone into `skills_root`. The user expects
dry-run to leave the filesystem unchanged outside ephemeral temp directories.

Implementation: when dry-run is set, clone into the same
`tempfile.TemporaryDirectory()` already used for dry-run snapshot generation.
The cloned repo is discarded with the temp dir at the end of the dry-run.

This allows dry-run to:

- Validate that the URL clones successfully (catch typos before real install).
- Resolve refs to commit SHAs and report what would be installed.
- Detect snapshot whitelist issues.

But never persists source repositories the user did not explicitly install.

## 8. Errors

| Condition | Exit | Message |
|---|---|---|
| Missing local source, no `git` field | per-project failure (1) | `Skill repository not found for <name>: <path>` (unchanged) |
| Missing local source, `git` present, clone fails | per-project failure (1) | `Failed to clone <name> from <git>: <git stderr>` |
| Local path exists but is not a git repository | per-project failure (1) | `Local skill path exists but is not a git repository: <path>` |

All errors mark the project as failed without halting other projects in an
`--all` run. Process exit code follows existing rules: `0` all succeeded, `1`
one or more failed.

## 9. Backwards Compatibility

- Existing `Skillfile.json` without `git`: unchanged behavior. `csk` fails
  with "not found" if the local repo is missing, same as v0.3.x.
- Existing local clones: `csk` uses them as before, regardless of whether
  `Skillfile.json` declares `git`. If the user manually cloned a skill from
  one URL and Skillfile lists a different `git`, the local clone wins. `csk`
  does not validate that the local remote URL matches.
- `csk update`: continues to fetch all repos under `skills_root` regardless
  of `git` field. Once a skill is cloned, it is a regular git repo with its
  own `origin`.
- Existing markers (`.csk-install.json`) without `git` field: remain valid.
  New installs record the `git` field when present in Skillfile, absent
  otherwise.

## 10. Marker Schema

`.csk-install.json` gains an optional `git` field recording the URL declared
in Skillfile at install time:

```json
{
  "schema_version": 1,
  "name": "skill-review",
  "source": "skill-review",
  "git": "git@gitlab.example.com:agents/skill-review.git",
  "ref_kind": "revision",
  "ref": "5ad3663...",
  "commit": "5ad3663...",
  ...
}
```

Field is absent when Skillfile declaration had no `git`. Useful for
`csk status` diagnostics and for verifying skill provenance post-install.

## 11. `csk update` Interaction

`csk update` continues to run `git fetch --all --tags --prune` for every
directory under `skills_root` that contains a `.git/`. The `git` field in
Skillfile is not consulted during `csk update`. Once a repo is cloned, it has
its own `origin` remote pointing at the URL; fetch uses that.

If Skillfile changes the `git` URL after a skill has been cloned from a
previous URL, `csk` does NOT reclone or rewrite the remote. The user must
either remove the local clone manually or run
`git remote set-url origin <new>`. Documented as a known limitation.

`csk update` does not clone missing repositories from Skillfile declarations.
It remains a fetch-only command over repositories already present under
`skills_root`. To bootstrap missing repositories, run `csk install` or
`csk upgrade`.

## 12. Test Surface

Required new tests under `tests/test_install.py`:

- `test_install_clones_missing_skill_from_git_url` — primary success path,
  asserts cloned repo exists under `skills_root` and marker records `git`.
- `test_dry_run_clones_git_url_only_to_temporary_location` — dry-run does
  not pollute `skills_root` or project files.
- `test_missing_skill_without_git_url_still_fails` — backwards-compat error
  preserved.

Recommended additional tests:

- `test_install_uses_existing_local_clone_even_when_git_declared` — local
  wins over remote, no reclone.
- `test_install_clone_failure_produces_clean_error` — invalid URL or auth
  failure reports cleanly without traceback.
- `test_local_path_exists_but_not_git_fails_cleanly` — covers point 4 of the
  resolution algorithm.
- `test_csk_update_fetches_cloned_skill_origin` — integration check that
  `csk update` works on auto-cloned skills.

Existing backwards-compat tests in `test_install.py` and `test_manifest.py`
remain green without modification.

## 14. Release Sequencing

1. Implementation lands on `main` as a single signed commit.
2. CHANGELOG entry under `[0.4.0]`.
3. Signed tag `v0.4.0` triggers `release.yml`.
4. PyPI publish + GitHub Release with artifacts and SHA256SUMS.
5. Homebrew formula bump in `ivanopcode/homebrew-csk`.
6. Distribution smoke runs automatically against the new release.
7. README and CHANGELOG updates land in the same commit as the
   implementation.

No deprecation cycle needed: the change is purely additive and backwards
compatible.

## 15. Out of Scope

Deferred to future RFCs:

- Source aliases via `~/.cocoaskills/config.json` (`skill_sources` map) for
  cases when many projects share the same skill URLs (RFC 0003 candidate).
- Public skill registry / discovery service (v1.0+).
- Mirror / fallback URL lists per skill.
- Authentication helpers built into `csk` (token storage, OAuth, etc.).
- Local-remote URL drift validation and automatic remote rewrite.
- Shallow clone optimization.
