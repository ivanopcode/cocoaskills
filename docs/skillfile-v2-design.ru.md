# CocoaSkills RFC 0007 — Skillfile v2: замыкание зависимостей, файл блокировки и поверхности

Статус: черновик — принят при условии редакционной правки
Реализация: можно начинать фундамент v0.9
Дата: 2026-06-30
Автор: Ivan Oparin
Цель: v0.9.0 (фундамент), в следующих выпусках — диапазоны/семантическое версионирование, настройка MCP

До RFC 0006 включительно `Skillfile.json` проекта был **плоским списком** независимо устанавливаемых навыков: без зависимостей навык→навык, без транзитивного разрешения, без порядка установки, без согласования версий и без обработки конфликтов. Единственным понятием зависимости был `type: system` — внешняя исполняемая программа, наличие которой проверяется через `shutil.which`.

Этот RFC задает слой зависимостей: манифест узла (`csk-skill.json` v4), в котором навык объявляет, что он предоставляет и что ему требуется; графовый манифест (`Skillfile.json` v2), который владеет составом проекта и политикой разрешения; детерминированный механизм разрешения с файлом блокировки; и три опоры безопасности — активные поверхности установки, безопасность контракта и интеграция с политикой источников. Главная забота здесь не арифметика версий, а предотвращение **скрытого расширения поверхностей**: команда не должна попасть в `PATH` без своих правил, контекст не должен попасть в окно без явного запроса, а транзитивный граф не должен подтянуть источник, которому корневой манифест на самом деле не доверял.

**В v0.9 разрешены только точные ссылки.** Диапазонные ограничения и выбор минимальной версии (MVS) описаны как целевая модель, но отложены до появления контракта семантического версионирования навыков (раздел 21). В v0.9 каждая зависимость указывает точную git-ссылку; разрешение — это **сведение точных ссылок**, а не подбор версий.

## 1. Цели

- Дать навыку возможность объявлять зависимости от: системной утилиты, команды другого навыка, контекста другого навыка и контракта MCP-сервера.
- Разрешать транзитивное замыкание проекта или рабочего процесса в детерминированную, воспроизводимую установку, записанную в файл блокировки.
- Делать каждую установленную поверхность — команды в `PATH`, контексты в окне, требования MCP — явной и пригодной для проверки; никогда не расширять ее неявно.
- Держать разрешение источников вне проекта, чтобы скомпрометированный манифест не мог сам выдать себе доверие.
- Сохранить текущее поведение для прямо объявленных плоских навыков: обратная совместимость.
- Разрешить композицию рабочих процессов (`includes`), чтобы рабочий процесс был переиспользуемым артефактом, а не скопированным списком навыков.

## 2. Не цели

- Реестр пакетов или служба размещения. Источники остаются git-URL; допуск определяется политикой.
- Диапазонные ограничения, математика семантического версионирования или MVS в v0.9: только точные ссылки, см. раздел 10.
- Проверка версий системных зависимостей в v0.9. v0.9 проверяет только наличие команды; будущие проверки версий должны использовать встроенные probe-механизмы, которыми владеет CocoaSkills, а не команды, аргументы, скрипты или post-install хуки из манифеста.
- Псевдонимы или пространства имен команд в v1, см. раздел 14.
- Навык, который поставляет собственный MCP-сервер как цель установки (`provides.mcp_servers`). Это будущая работа, раздел 17, и отдельный RFC, потому что такой режим заставляет `csk` писать настройки адаптеров агента.
- Формальный контракт семантического версионирования навыка. Он нужен до диапазонов и будет описан в под-RFC, см. раздел 21.

## 3. Предыстория и текущее состояние

- `Skillfile.json`: `skills: [{ name, git, tag|branch|revision }]`; каждый навык закреплен точной git-ссылкой. Установка воспроизводима, диапазонов нет.
- `csk-skill.json`: `commands` смешивает `type: script` — команду, которую навык **предоставляет**, — и `type: system` — внешнюю утилиту, которая навыку **требуется**. Схема v2 добавляет `runtime_roots`; схема v3 добавляет оболочку `capabilities`, используемую `csk audit`.
- Установщик проходит по плоскому списку `skills`: без графа, порядка и транзитивного шага.
- Обертки команд находятся в **плоском пространстве имен**: `.agents/bin/<command>` для проекта, глобальное зеркало; приоритет: project > global > system.
- Хранилище среды выполнения **адресуется по содержимому через commit**: `~/.cocoaskills/runtime/<skill>/<commit>/`.
- Уже существуют два установочных слоя, и они здесь центральные: **контекст подсказки** (`whitelist.INCLUDE_ROOTS` → `.agents/skills/<skill>/`) и **среда выполнения** (`runtime_roots` → хранилище среды выполнения + обертки команд). Типы зависимостей в этом RFC напрямую отображаются на эти два слоя.
- `audit/source_policy.py` уже существует, но сегодня он классифицирует источник как internal/public для *облачного аудита* и нормализует git-источники в основном до **хоста** (раздел 13).
- `dependencies.json` — устаревший файл: он копируется в контекст, но никогда не разбирается. Он удаляется отдельной задачей и заменяется блоком `dependencies`, описанным здесь.
- MCP сегодня нигде в `csk` не упоминается.

## 4. Мотивирующий пример: случай `wk`

Такой пример уже есть в реальных навыках, но пока спроектирован наполовину:

- `skill-wiki` предоставляет команду `wk` (`type: script`) и требует внешнюю CLI-утилиту `wiki` (`type: system`).
- `skill-wiki-memory` содержит недокументированный блок, объявляющий, что ему нужен `wk` из `skill-wiki`. `csk` никак на это не реагирует: подсказка просит человека вручную добавить `skill-wiki`. Уже этот пример показывает основной вопрос: зависимость здесь от *команды* `wk` (среда выполнения) или от *контракта* `skill-wiki` (контекст)? При этом `skill-wiki-memory` объявляет `capabilities.exec: "none"`, хотя вызывает `wk`; этот RFC делает такую несогласованность недопустимой.

## 5. Термины

- **Узел** — навык, описанный в `csk-skill.json`. Объявляет предоставляемое, требуемое и возможности по **имени и ссылке**, но никогда по источнику.
- **Графовый манифест** — `Skillfile.json`. Владеет прямыми навыками, включениями, источниками, переопределениями и создает файл блокировки. Корневой графовый манифест — источник истины.
- **Замыкание** — транзитивное множество узлов, достижимых из корня через включения и зависимости узлов.
- **Поверхность** — то, что материализует установка: команды в `PATH`, контексты подсказки в окне, требования MCP.
- **Активация** — решение на конкретном ребре материализовать определенную поверхность поставщика. Это свойство **ребра**, а не узла.
- **Идентичность источника** — канонический `host/namespace/project` git-артефакта, навыка или включения, одинаково нормализованный для разных способов доступа.

## 6. Обзор архитектуры

Два манифеста, четкое разделение владения:

- **`csk-skill.json` (узел, схема v4)** — предоставляемое + требуемое + локальные возможности. Узел говорит, *что* ему нужно, но никогда — *откуда* это брать.
- **`Skillfile.json` (граф, схема v2)** — прямые навыки, `includes`, `sources` (имя → git URL), `overrides`, `agents`, `locale`. Разрешает замыкание и записывает `Skillfile.lock`.

Механизм разрешения строит замыкание, **разрешает точные ссылки в v0.9** (позже — выбирает версии через MVS, когда появится контракт семантического версионирования), проверяет конфликты и политику источников, записывает файл блокировки и устанавливает из него через два уже существующих установочных слоя.

## 7. Манифест узла — схема `csk-skill.json` v4

Физически `commands` не меняется, чтобы сохранить обратную совместимость: это по-прежнему поверхность **предоставляемого**. Все **требуемое** живет в `dependencies`.

```jsonc
{
  "schema_version": 4,
  "runtime_roots": ["scripts"],
  "capabilities": { /* только локальный узел; оболочка v3 */ },

  // ПРЕДОСТАВЛЯЕТ — экспортируемые команды (обертки)
  "commands": {
    "wk": { "type": "script", "unix_path": "scripts/wk", "win_path": "scripts/wk.cmd" }
  },

  // ТРЕБУЕТ
  "dependencies": {
    // требования к командам — system и skill используют одну модель; отличается только поставщик
    "commands": {
      "wiki": { "type": "system", "command": "wiki", "hint": "..." },
      "wk":   { "type": "skill", "skill": "skill-wiki", "command": "wk" }
    },
    // объявления узлов-навыков — точная ссылка + намерение по контексту (каноническая запись на поставщика)
    "skills": {
      "skill-wiki": { "ref": { "kind": "tag", "value": "v1.4.2" }, "context": { "mode": "full" } }
    },
    // требования MCP — контракт поставщика, а не просто имя сервера
    "mcp_servers": {
      "wiki-mcp": { "tools": ["article.search", "article.read"], "required": true, "hint": "..." }
    }
  }
}
```

Правила:

- `commands.*.type=system` — требование, которое живет в блоке предоставляемых команд, — **устарело, но допускается** на время миграции. Новые системные утилиты описываются в `dependencies.commands` с `type: system`.
- `dependencies.commands.*.type=system` в v0.9 объявляет **только наличие** команды: `command` и необязательный `hint`. Наличие проверяется через `shutil.which`; `csk` не выполняет проверки версий из манифеста. Ограничения версий системных утилит отложены. Когда они появятся, они должны использовать встроенные probe-id CocoaSkills, например `version_probe: "builtin:git"`, `version_probe: "builtin:python"` или `version_probe: "builtin:glab"`: манифест может назвать разрешенный probe-id, но не может задавать аргументы команд, shell-фрагменты, `check`, `install`, `post_install` или произвольные скрипты.
- `dependencies.commands.*` **не содержит git URL**. Запись `type: skill` ссылается на поставщика только по имени.
- `dependencies.skills.<provider>` — **каноническое объявление узла**: здесь находится **ref** и намерение по контексту. **Каждое требование команды с `type: skill` ОБЯЗАНО иметь соответствующую запись `dependencies.skills`.** Ссылка живет только здесь.
- `ref` — точная ссылка: `{ "kind": "tag" | "revision", "value": "..." }`. **Ребра зависимостей (`dependencies.skills.*`) в v0.9 принимают только `tag` или `revision`.** Диапазон семантического версионирования (`version: "^1.0.0"`) является ошибкой разбора до появления контракта семантического версионирования навыков (раздел 21). `branch` принимается только как устаревшее или сокращенное объявление в корне (раздел 8), но никогда на ребре зависимости: ветка точна как имя входной ссылки, но не неизменяема, поэтому разрешение записывает конкретный commit в файл блокировки, а `--locked` устанавливает именно этот commit и никогда не следует за сдвинутой веткой.
- `context` — объект: `{ "mode": "full" | "none", ... }`, с заделом под `{ "mode": "partial", "refs": [...] }` позже (раздел 12).
- `dependencies.json` не является частью схемы v4.

## 8. Графовый манифест — схема `Skillfile.json` v2

```jsonc
{
  "schema_version": 2,
  "project": { "alias": "incident-management" },
  "agents": ["claude_code", "codex_cli"],
  "locale": "ru",

  // адресная книга: имя -> git URL. НЕ выдает доверие (раздел 13).
  "sources": {
    "skill-wiki":   { "git": "git@gitlab.wildberries.ru:portals/agentic-infra/skills/skill-wiki.git" },
    "skill-gitlab": { "git": "git@gitlab.wildberries.ru:portals/agentic-infra/skills/skill-gitlab.git" }
  },

  // прямые навыки: точная ссылка; источник берется из `sources`
  "skills": [ { "name": "skill-gitlab", "ref": { "kind": "tag", "value": "v1.1.0" } } ],

  // композиция рабочий процесс -> рабочий процесс (буквально «транзитивный Skillfile.json»)
  "includes": [
    { "name": "incident-management",
      "git": "git@gitlab.wildberries.ru:portals/agentic-infra/workflows/incident-management.git",
      "ref": { "kind": "tag", "value": "v1.0.0" } }
  ],

  // корень может закрепить точную ссылку узла; допустимо только если она разрешается в commit,
  // совместимый с точными ограничениями узла (раздел 10)
  "overrides": { "skill-wiki": { "ref": { "kind": "tag", "value": "v1.4.2" } } }
}
```

- Прямой навык — это объявление с **точной ссылкой/ref или revision**, разрешаемое через `sources`. Устаревшие ключи `tag`/`branch`/`revision` на уровне навыка остаются допустимым сокращением; устаревшее поле `skills[].git` принимается и нормализуется как неявная корневая запись `sources[skill.name].git`. Новые манифесты v2 ДОЛЖНЫ использовать `sources`; если `skills[].git` дублирует `sources[name]`, значения должны совпасть, иначе это ошибка. Ссылка на ветку разрешается в зафиксированный commit (раздел 7); `--locked` никогда не следует за сдвинутой веткой. Слово *constraint* используется в общем смысле; в v0.9 ограничение всегда является точной ссылкой, а не диапазоном.
- `includes` добавляют **только навыки и ограничения**. Корень владеет `sources`, `overrides`, `agents`, `locale` и единственным файлом блокировки. Поля политики из включенных манифестов игнорируются (раздел 13.4).

## 9. Классификация зависимостей

| Вид | Смысл | Где объявляется | Установочный слой |
|------|---------|-------------|---------------|
| D1 — системная утилита | нужна внешняя исполняемая программа в `PATH` | `dependencies.commands.<n>.type=system` | только проверка, никогда не устанавливается |
| D2 — команда навыка (среда выполнения) | нужна экспортируемая команда другого навыка | `dependencies.commands.<n>.type=skill` + парная `dependencies.skills` | слой среды выполнения (обертка команды) |
| D3 — контекст навыка | нужны SKILL.md/правила другого навыка в окне | `dependencies.skills.<p>.context.mode=full` | слой контекста подсказки |
| D4 — контракт MCP | нужны MCP-сервер и инструменты в сеансе | `dependencies.mcp_servers.<s>` | только проверка/поверхность (T1) |

D2 и D3 — **ортогональные оси** над одним и тем же поставщиком. Навык может требовать только команду (среду выполнения), только контекст или и то и другое. Сторона поставщика для D2 — это просто его `commands` (`type: script`).

## 10. Модель разрешения

- **Замыкание.** Начиная с корневого графового манифеста, рекурсивно разворачивать `includes` и `dependencies.skills` / `dependencies.commands(type=skill)` каждого узла до неподвижной точки.
- **v0.9 — это сведение точных ссылок, а не выбор версий.** Алгоритм для каждого имени навыка:
  1. собрать все входящие ограничения, каждое из которых является точной ссылкой;
  2. разрешить каждую ссылку в конкретный commit относительно `source` навыка;
  3. если все они разрешаются в **один и тот же commit** → OK;
  4. если они разрешаются в **разные commit** → конфликт (ошибка);
  5. корневой `override` совместим **тогда и только тогда, когда он разрешается в тот же commit, что и каждая входящая точная ссылка**; будущая семантика диапазонов/MVS может определить более широкое понятие совместимости.
  Пример: `A → skill-wiki tag v1.4.2` и `B → skill-wiki revision abc123` **совместимы тогда и только тогда, когда `v1.4.2` разрешается в `abc123`**, иначе это конфликт. Файл блокировки записывает **все** входящие ограничения и выбранный commit.
- **Целевая модель, отложена.** Когда появится контракт семантического версионирования, шаги 1–5 заменяются диапазонами и выбором минимальной версии; механика файла блокировки и `--locked` не меняется.
- **Разрешение источника.** Узел называет зависимости по имени навыка; URL берется из корневой карты `sources` (раздел 13), а не из ребра зависимости. Нет источника → разрешение завершается ошибкой.
- **Поставщик действительно предоставляет команду.** Ребро `dependencies.commands.<n>.type=skill` допустимо только если `commands` выбранного поставщика содержит запрошенную команду с `type: script`. Отсутствующая команда у поставщика — ошибка разрешения; для команды, которую поставщик не экспортирует, не строится ни замыкание, ни файл блокировки, ни поверхность.
- **Порядок.** Топологическая сортировка, чтобы поставщики устанавливались до потребителей. Цикл — жесткая ошибка.
- **Файл блокировки.** Разрешение записывает `Skillfile.lock` (раздел 18): для каждого узла — точный commit, хеши, активные поверхности, идентичность источника, входящие ограничения и доверенный закрепленный хеш.
- **Установка из файла блокировки, с явным закрытым режимом.** `csk resolve` — единственная команда, которая записывает файл блокировки. `csk install --locked` устанавливает строго из файла блокировки и **завершается ошибкой, если файла нет или он устарел**; это воспроизводимый путь для CI. Обычный `csk install` может создать **отсутствующий** файл блокировки при первом запуске; если файл блокировки **существует, но устарел**, команда **завершается ошибкой с понятным действием** — запустить `csk resolve`, — и никогда не выполняет тихое повторное разрешение существующего файла. Воспроизводимость не должна зависеть от того, где была запущена команда, поэтому CI использует `--locked`. `csk upgrade` = fetch + `csk resolve` + install для старого пользовательского опыта; `csk update` сохраняет текущее значение «только fetch» и не записывает файл блокировки.

Почему точные ссылки + файл блокировки, а не диапазоны: это сохраняет воспроизводимость CocoaSkills, пока не требует решателя версий или контракта семантического версионирования и делает обновления намеренными. Диапазоны без файла блокировки и строгой политики источников явно отвергаются.

## 11. Опора 1 — активные поверхности установки

Конфликты и раскрытие считаются по **активированным** поверхностям, а не по всему, что узел мог бы экспортировать.

- **Активация — свойство ребра.** Один и тот же поставщик может быть активирован разными потребителями по-разному. Эффективная активная поверхность узла — это **объединение по всем входящим ребрам**: корень + каждый потребитель.
- **Значения по умолчанию.** **Прямой** навык из корневого `skills` активирует `context: full` + **все** свои команды: это сохраняет текущее поведение. **Транзитивная зависимость среды выполнения** активирует только **явно запрошенные** команды. **Транзитивная контекстная зависимость** активирует только контекст подсказки, **без** командных оберток.
- **Приоритеты объединения.**
  - Эффективная поверхность **контекста**: `full` побеждает `none`. Позже для partial: объединение `refs`, если ни одно ребро не просит `full`.
  - Эффективная поверхность **команд**: объединение явно активированных имен команд.
- **Коллизия** проверяется по объединению **активных командных оберток**: для каждого имени команды в замыкании может быть активна не более одной пары `(skill, commit)`. Команда, которую узел экспортирует, но которую никто не активирует, никогда не записывается в `.agents/bin` и никогда не конфликтует.
- **Граница детализации.** Активация управляет **командной оберткой** (`PATH`) и **контекстом** (окно). **Хранилище среды выполнения целиком копируется на версию навыка**: активация любой одной команды копирует все `runtime_roots`. Это приемлемо, потому что хранилище вне контекста; изоляции среды выполнения по отдельным командам нет.
- **Файл блокировки хранит активные поверхности** по каждому узлу: `{ context, commands, mcp }` плюс `activation_edges`, которые их породили. Эффективная поверхность — объект первого класса, пригодный для проверки: `csk audit` и проверяющий видят до установки, что именно попадает в `PATH`, окно и MCP.

## 12. Опора 2 — безопасность контракта

Главный инвариант: **обе оси включаются явно; ни одна не подразумевает другую; `csk` материализует только то, что явно активировано.** Зависимость от команды никогда автоматически не подтягивает контекст; зависимость от контекста никогда автоматически не устанавливает команды.

- Каждая запись `dependencies.commands.*.type=skill` ОБЯЗАНА иметь парную `dependencies.skills.<provider>`; ссылка живет только в `dependencies.skills`.
- Объект `context`:
  - `{ "mode": "full" }` — подтянуть SKILL.md и правила поставщика в контекст потребителя; контракт принадлежит поставщику.
  - `{ "mode": "none", "contract": "consumer", "refs": ["SKILL.md", "references/wiki-usage.md"] }` — только среда выполнения; потребитель берет ответственность на себя и указывает, где он документирует безопасное использование заимствованной команды. `contract` имеет смысл только при `mode: none`.
- **Контракт проверяется ПО КАЖДОМУ ребру среды выполнения, а не по эффективной поверхности поставщика.** Если потребитель A использует `wk` с `mode: none` и без потребительского контракта, A небезопасен, даже если другой потребитель B случайно подтягивает `skill-wiki` с `mode: full`. Каждое ребро среды выполнения должно явно выбрать: контракт поставщика (`mode: full`) или контракт потребителя (`mode: none` + `contract` + `refs`). Никакой «безопасности по счастливой случайности».
- **Инструмент без контракта.** Ребро зависимости команды с `type: skill`, `context.mode=none` и без потребительского контракта → **предупреждение в обычном режиме, ошибка в строгом аудите**. Устраняется через `mode: full` или через `contract: consumer` + `refs`.
- **Многоуровневая проверка.** Статический минимум: файлы из `refs` существуют и попадают в контекст потребителя через `INCLUDE_ROOTS`. Семантическая проверка: LLM/статический аудиторский модуль (RFC 0006) проверяет, что эти ссылки действительно описывают безопасное использование. Статическая проверка выполняется всегда; семантическая — только через аудиторские модули. Статическая проверка гарантирует наличие артефакта, а не его смысл, и прямо об этом сообщает.
- **Область действия.** Безопасность контракта применяется только к зависимостям команд навык→навык. `type: system` сохраняет модель «объяви и дай подсказку».
- **Контекст без инструмента.** `context.mode=full` для поставщика, который экспортирует команды, но не имеет **ни одной** активированной команды → **предупреждение**: «контекст описывает команды, но обертки не установлены». В v0.9 выводится только предупреждение; `csk` никогда не устанавливает команды автоматически. Схема оставляет место для явного подавления через намерение: `"commands": { "mode": "none" }` рядом с `context` или `runtime: "none"` в записи `dependencies.skills.<provider>`, чтобы контекстные зависимости от методологических навыков или навыков только с правилами не оставались шумными. В v0.9 это не требуется.

## 13. Опора 3 — интеграция с политикой источников

Граф не должен уметь отмывать доверие. **Адрес — не разрешение.**

### 13.1 Роли

- `sources` в корневом `Skillfile.json` — это **адресная книга**: имя → git URL. Она не выдает доверие.
- `source_policy` — **единственный механизм допуска**, и он живет **вне проекта**: в настройках машины или организации. Поэтому скомпрометированный `Skillfile.json` не может сам разрешить вредоносный источник.

### 13.2 Идентичность с учетом пути: единая модель git-артефакта

Сегодня `audit/source_policy.py` нормализует git-источники в основном до **хоста**, а это слишком грубо: `portals/agentic-infra/skills/*` и любой другой проект на том же GitLab имеют разные границы доверия. Механизму разрешения нужна каноническая идентичность **хост + пространство имен + проект**, единая для разных способов доступа:

```text
git@gitlab.wildberries.ru:portals/agentic-infra/skills/skill-wiki.git
https://gitlab.wildberries.ru/portals/agentic-infra/skills/skill-wiki
        -> id: gitlab.wildberries.ru/portals/agentic-infra/skills/skill-wiki
```

Нормализация: привести хост к нижнему регистру, убрать завершающее `.git`, убрать начальный `/`, сохранить регистр пути, игнорировать SSH-пользователя и способ доступа `ssh://`/`https://`/`git@`. Это единая модель **источника git-артефакта**: репозитории навыков и `includes` — репозитории рабочих процессов — нормализуются одной функцией и допускаются одной политикой. Общий нормализатор живет в `csk.source_identity`; две политики, которые его используют, остаются раздельными (раздел 13.5).

### 13.3 Схема политики источников и приоритеты

Шаблоны сопоставляются с **полным каноническим id, включая хост**. `default` задает поведение закрытого мира.

```jsonc
"source_policy": {
  "default": "deny",
  "rules": [
    { "effect": "allow", "pattern": "gitlab.wildberries.ru/portals/agentic-infra/skills/*" },
    { "effect": "allow", "pattern": "gitlab.wildberries.ru/portals/agentic-infra/workflows/*" },
    { "effect": "deny",  "pattern": "gitlab.wildberries.ru/portals/agentic-infra/skills/experimental/*" }
  ],
  "revocations": [
    "gitlab.wildberries.ru/portals/agentic-infra/skills/compromised"
  ]
}
```

Семантика glob-шаблонов: `*` соответствует одному сегменту пути, `**` — рекурсивно. **Приоритеты по порядку:**

1. **Отзыв всегда является жестким запретом** и не может быть переопределен никаким allow: отзыв по безопасности — не обычное правило, которое можно обойти более конкретным allow.
2. Среди `rules` побеждает **самый конкретный** шаблон.
3. При равной конкретности **deny побеждает**, а если ничего не совпало, применяется `default`.

### 13.4 Включения и источники

- Включенный `Skillfile.json` добавляет **только навыки и ограничения**. Источники (`sources`) предоставляет корень. Нет корневого источника для включенного навыка → механизм разрешения **падает**; скрытого импорта источников нет.
- **Собственный источник включения** — репозиторий рабочего процесса — сам является источником git-артефакта: он допускается через `source_policy` на этапе разрешения, заново проверяется при установке (раздел 13.5) и записывается в файл блокировки (раздел 18). Включение может прийти по ref и изменить ограничения, а значит и замыкание; поэтому это часть цепочки поставки, а не доверенный короткий путь.
- Аварийный выход, позже: `import_sources: true` на включении, под контролем `source_policy`. По умолчанию выключено. Цена: чтобы принять переиспользуемый рабочий процесс, корень должен знать его источники или явно импортировать их. Это сознательная цена за отказ доверять заявлениям источников внутри включения.

### 13.5 Допуск, повторная проверка и две политики

- **Допуск выполняется до любой сетевой git-операции.** Механизм разрешения проверяет идентичность источника до любого clone/fetch/archive навыка или включения; недопущенные источники никогда не клонируются, не скачиваются и не анализируются. Отзывы и правила применяются ко всему замыканию: навыкам и включениям. Установка заново проверяет зафиксированные источники перед материализацией.
- **`csk install` выполняет повторную проверку.** Установка заново проверяет каждый источник из файла блокировки по **текущей** политике; если источник был отозван после записи файла блокировки, установка падает даже при действительном файле блокировки.
- Записанный в файле блокировки `matched_rule` — **только диагностическое поле, не разрешение**. Установка всегда заново проверяет текущую политику; файл блокировки никогда не выдает доверие.
- Две отдельные политики разделяют только нормализатор: `source_policy.authorization` — allow/revoke для установки, и `audit.source_policy.classification` — internal/public для облачного аудита. Они называются отдельно, чтобы не смешивать границы безопасности.

## 14. Конфликты

- **Одно имя навыка на замыкание** → ровно один commit во всем замыкании.
- **Одна активная команда на `.agents/bin`.** Два разных навыка, активирующих одно и то же имя команды (`wk`, `mr`), → ошибка.
- **Конфликт ограничений → ошибка**, а не молчаливый победитель. В v0.9 это случай разных commit из раздела 10; разные major-версии команды означали бы другой интерфейс, из-за чего инструкции в SKILL.md потребителя стали бы неверными, поэтому они никогда не объединяются молча.
- **Корневой override** разрешен только если он разрешается в совместимый commit (раздел 10).
- **Псевдонимов команд в v0.9 нет.** Псевдонимы (`skill-wiki/wk`) заставили бы инструкции в SKILL.md расходиться с реальным `PATH`. Имена команд остаются стабильными и совпадают с тем, на что ссылается SKILL.md. Документированный путь повторного экспорта можно добавить позже без ломающего изменения.

## 15. Возможности: локальный узел и эффективное замыкание

- **Локальные** `capabilities` описывают один узел. Проверка согласованности локальна и статична: локально объявленные командные зависимости узла (`dependencies.commands.*` — `wk` / `wiki` / `glab`, которые он вызывает) должны быть подмножеством локального `capabilities.exec` этого узла. `dependencies.skills.*` (контекст) не обязательно означает `exec`.
- **Эффективные** возможности = **объединение локальных возможностей по всему замыканию**. `csk plan` / `csk audit` вычисляет и показывает их, чтобы проект видел, что может делать его полное замыкание.
- Различие важно: строгая проверка — это **локально объявленные зависимости против локально объявленного exec**, а не зависимости против эффективной поверхности; второе было бы самоподтверждающимся. **Строгий режим падает, когда локальные зависимости узла расходятся с его локальными возможностями**: случай `skill-wiki-memory exec:none + wk`.
- **MCP и возможности (v0.9 = вариант B):** требования MCP живут в `dependencies.mcp_servers` и только в эффективной поверхности замыкания; v0.9 **не** добавляет поле `capabilities.mcp`. Сетевой выход, если он известен, все равно должен отражаться в `capabilities.network`.
- Строгий аудит применяется к **каждому узлу в замыкании**, по поколению схемы:
  - **v1 / v2** без capabilities: строгий режим требует хеш-закрепление содержимого или миграцию.
  - **v3** с capabilities, но без графа зависимостей: совместима со строгим режимом как **лист / поставщик**, но не может объявлять новый граф зависимостей.
  - **v4**: полноценный узел зависимостей.

## 16. Взаимодействие областей: проектная и глобальная

- Зависимости по умолчанию разрешаются **локально для проекта**; командной зависимости нужен конкретный commit, а опора на потенциально другой глобальный навык нарушила бы воспроизводимость.
- Глобальные навыки остаются пользовательским удобством и **никогда не становятся скрытым поставщиком** для замыкания проекта. Файл блокировки всегда закрепляет проектно-локальный узел. Это покрывается тестами (раздел 25), а не только текстом.

## 17. MCP

- Моделируется как **контракт поставщика**: `dependencies.mcp_servers.<name> = { tools: [...], required, hint }`. В v0.9 это не поле `capabilities.mcp` (раздел 15).
- T1, этот RFC: объявляется и показывается; не исполняется принудительно, потому что `csk` не может проверить MCP-инструмент через `shutil.which`.
- T2, спроектировано на будущее: если настройки MCP агента можно обнаружить, предупреждать об отсутствующем обязательном сервере или инструменте.
- T3, отложено в отдельный RFC: навык поставляет MCP-сервер (`provides.mcp_servers`), а `csk` записывает настройки адаптеров. Это новая цель установки со своими последствиями для доверия и сети.

## 18. Файл блокировки — `Skillfile.lock`

Файл коммитится рядом с `Skillfile.json`: намерение + разрешение. Сгенерированное `.agents/**` по-прежнему не коммитится. Файл блокировки — объект API первого класса, а не кэш. Хеши разделены, чтобы всегда было понятно, *что именно* было хешировано, а политика **не** входит в хеш разрешения: безобидная правка allow-правила не должна делать файл блокировки устаревшим; установка отдельно заново проверяет политику.

```jsonc
{
  "schema_version": 1,
  "resolver_version": "0.9.0",

  // проверка устаревания: корневой Skillfile + включенные Skillfile + адреса источников + точные ссылки.
  // политика сюда НЕ входит.
  "resolution_input_hash": "<sha256>",
  // только диагностика, опционально; никогда не используется для доверия или устаревания
  "policy_snapshot_hash": "<sha256>",

  "includes": {
    "incident-management": {
      "ref": { "kind": "tag", "value": "v1.0.0" },
      "commit": "<full-sha>",
      "manifest_path": "Skillfile.json",
      "manifest_hash": "sha256:<hash of the included Skillfile only>",
      "snapshot_hash": "sha256:<optional full repo snapshot>",
      "source": { "id": "gitlab.wildberries.ru/portals/agentic-infra/workflows/incident-management",
                  "matched_rule": "allow gitlab.wildberries.ru/portals/agentic-infra/workflows/* (diagnostic)" }
    }
  },

  "nodes": {
    "skill-wiki": {
      "commit": "<full-sha>",
      "hashes": {
        "snapshot": "sha256:<full git-archive snapshot hash, what audit/trust verifies>",
        "context":  "sha256:<installed prompt-context hash, what the install marker verifies>",
        "runtime":  "sha256:<runtime_roots hash, optional>"
      },
      "source": { "id": "gitlab.wildberries.ru/portals/agentic-infra/skills/skill-wiki",
                  "matched_rule": "allow gitlab.wildberries.ru/portals/agentic-infra/skills/* (diagnostic)" },
      "pulled_by": [ { "from": "skill-wiki-memory", "ref": { "kind": "tag", "value": "v1.4.2" } } ],
      "activation_edges": [ { "from": "skill-wiki-memory", "context": "none", "commands": ["wk"] } ],
      "surfaces": { "context": "none", "commands": ["wk"], "mcp": [] },
      "trust": { "pin": "sha256:<...>" }
    }
  }
}
```

Поузловые `hashes` разделяют три хеша, которые реализации иначе склонны смешивать: `snapshot` — то, что проверяет аудит/доверие; `context` — то, что исторически проверяет установочный маркер; и необязательный `runtime`. `matched_rule` — диагностическое поле; установка заново проверяет текущую политику (раздел 13.5).

## 19. Миграция и обратная совместимость

- **Прямые навыки не меняются.** Плоский `Skillfile.json` с точными git-ссылками продолжает работать; устаревшие ключи `tag`/`branch`/`revision` принимаются как сокращение для `ref`. Прямой навык по умолчанию активирует `context: full` + все команды.
- **Старые схемы узлов устанавливаются как листья.** `csk-skill.json` v1/v2/v3 не имеют блока `dependencies`. Строгое поведение по поколению: v1/v2 требуют хеш-закрепление содержимого или миграцию; **v3 остается совместимой со строгим режимом как лист/поставщик**, но не может объявлять ребра зависимостей; v4 — полноценный узел (раздел 15).
- **`commands.type=system`** остается устанавливаемым, но устаревшим; `csk` может предупреждать и указывать на `dependencies.commands`.
- **`dependencies.json` удален** отдельной задачей.
- **Внедрение файла блокировки.** `csk resolve` — единственная команда, которая записывает файл блокировки. `csk install --locked` требует свежий файл блокировки: путь для CI; обычный `csk install` может создать отсутствующий файл блокировки при первом запуске, но **падает на устаревшем файле блокировки** с подсказкой запустить `csk resolve`. Расхождение между `Skillfile.json` / включенными Skillfile и файлом блокировки, сравниваемое через `resolution_input_hash`, является сигналом для `csk status`. Одно только изменение политики не делает файл блокировки устаревшим: установка отдельно заново проверяет политику. `csk update` сохраняет значение «только fetch»; `csk upgrade` = fetch + resolve + install.

## 20. Сквозной пример

`workflows/incident-management` (корень) объявляет `skill-gitlab`. Добавляем долговременную память:

1. Корневой `Skillfile.json` добавляет `skill-wiki-memory` на `ref tag v2.1.0` и запись `sources` для него.
2. Манифест узла `skill-wiki-memory` объявляет `dependencies.skills.skill-wiki` на `ref tag v1.4.2` с `context.mode = none, contract: consumer, refs: ["references/wiki-usage.md"]`, а также `dependencies.commands.wk = { type: skill, skill: skill-wiki, command: wk }`.
3. Замыкание механизма разрешения: `{ skill-gitlab, skill-wiki-memory, skill-wiki }`. Источник `skill-wiki` должен быть в корневом `sources` — иначе ошибка — и пройти `source_policy`.
4. Сведение точных ссылок: `skill-wiki` разрешает `tag v1.4.2 → commit abc123`; других ограничений нет; выбранный commit — `abc123`. Топологический порядок ставит `skill-wiki` перед `skill-wiki-memory`.
5. Активные поверхности: `skill-wiki` активируется **только как среда выполнения** с командой `wk`, без контекста, потому что потребитель выбрал `mode: none` и предоставил потребительский контракт. Безопасность контракта проходит по ребру: файлы из `refs` присутствуют; семантическая проверка выполняется через аудиторские модули.
6. Файл блокировки записывает все узлы — с `hashes`, входящими ссылками, `activation_edges`, поверхностями и закрепленными хешами доверия, — включение и идентичности источников.
7. Установка: обертка `wk` попадает в `.agents/bin`; SKILL.md от `skill-wiki` **не** добавляется в контекст; собственный SKILL.md `skill-wiki-memory` с `references/wiki-usage.md` несет контракт безопасного использования.

Итог: шаблон памяти использует `wk` с явным, проверяемым контрактом; полный контекст `skill-wiki` никогда не попадает в окно скрыто.

## 21. Открытые вопросы и под-RFC

- **Контракт семантического версионирования навыков**: что major/minor/patch означают для интерфейса экспортируемой команды. Пока этого контракта нет, **диапазонные ограничения не принимаются; v0.9 принимает только точные теги или точные revision; файл блокировки записывает разрешенный commit и хеш содержимого.** Диапазоны + MVS появятся вместе с контрактом.
- **Канал принудительного повышения по рекомендациям безопасности**: как поднимать известную уязвимую транзитивную ссылку при exact-only/MVS, с привязкой к слою аудита/доверия; файл блокировки должен нести закрепленный хеш безопасности.
- **Проверки версий системных зависимостей**: как объявлять требования вроде `glab >= 1.50.0`, не позволяя skill-манифесту выполнять произвольные проверки. Целевое направление — enum встроенных probe-id CocoaSkills (`builtin:git`, `builtin:python`, `builtin:glab` и т.д.), а не аргументы или скрипты из манифеста.
- **Распространение политики источников**: как политика организации попадает на машину и как `source_policy.py` вырастает от гранулярности хоста до идентичности с учетом пути из раздела 13.2.
- **Частичный контекст** (`context.mode = partial`, `refs`): подтягивание только отдельных ссылок поставщика, чтобы беречь окно контекста локальной модели.
- **MCP T3**: `provides.mcp_servers` как цель установки.
- **Повторный экспорт команд / пространства имен** после плоской модели v1, которая сразу падает при конфликте.

## 22. Фундаментальный срез v0.9

Этот RFC охватывает фундамент и несколько крупных будущих блоков. В v0.9 поставляется только фундамент.

В v0.9:

- Разбор схемы `Skillfile.json` v2: `sources`, `skills`, `includes`, `overrides`. **Только точные ссылки**; диапазоны отклоняются (раздел 10).
- Разбор схемы `csk-skill.json` v4: `dependencies.commands`, `dependencies.skills`, `dependencies.mcp_servers`; старые схемы — как листья (раздел 19).
- **Политика источников с учетом пути**: идентичность и семантика allow/deny/revoke (раздел 13). Блокирующая часть, не опция.
- Построитель замыкания — сведение точных ссылок — и модель активации активных поверхностей (разделы 10, 11).
- Модель файла блокировки с `resolver_version`, разделенными хешами, `includes`, `activation_edges` (раздел 18).
- Установка из файла блокировки поверх сегодняшних примитивов среды выполнения/контекста, с `--locked` (раздел 10).
- Безопасность контракта только со **статической** проверкой `refs`; семантическая проверка — улучшение аудита, а не блокирующее условие v0.9 (раздел 12).
- Только MCP T1: поверхность метаданных без принудительного исполнения, вариант B для возможностей (разделы 15, 17).
- Сохранена совместимость с прямыми плоскими навыками (разделы 11, 19).

Отложено, каждое в свой выпуск/RFC: диапазонные ограничения + контракт семантического версионирования навыков; проверки версий системных зависимостей через встроенные probe CocoaSkills; канал принудительного повышения по рекомендациям безопасности; MCP T2/T3; частичный контекст; повторный экспорт команд / пространства имен; `import_sources` для включений.

## 23. Решения для v0.9, ранее открытые

- **D-1 — только точные ссылки.** Никакого временного поиска «самого высокого подходящего тега»: это псевдосемантическое версионирование без контракта.
- **D-2 — режимы установки.** `csk resolve` — единственный писатель файла блокировки; `csk install --locked` строгий; обычный `csk install` может создать отсутствующий файл блокировки при первом запуске, но **падает на устаревшем файле блокировки**. `csk update` остается fetch-only; `csk upgrade` = fetch + resolve + install. CI и обертки используют только `--locked`.
- **D-3 — кэш включений — соседнее хранилище**, а не `skills_root`, потому что включение рабочего процесса не является устанавливаемым навыком: `~/.cocoaskills/cache/workflows/<source-id-hash>/<commit>/snapshot/`. Файл блокировки хранит `ref`, `commit`, `manifest_hash` и id источника включения.
- **D-4 — прямая корневая активация** записывается как синтетическое ребро; `["*"]` раскрывается в конкретный список команд в `surfaces.commands` при материализации:

  ```jsonc
  "activation_edges": [ { "from": "<root>", "reason": "direct", "context": "full", "commands": ["*"] } ]
  ```

- **D-5 — глобальная область никогда не поставщик** — это закрепляется тестами, а не текстом: транзитивная зависимость от глобально установленного навыка все равно требует корневую запись `sources` и проектно-локальный узел в файле блокировки.
- **D-6 — вывод MCP T1.** `csk status`: сводка требований; `csk audit --json`: часть эффективной поверхности замыкания:

  ```jsonc
  "mcp": [ { "node": "skill-wiki-memory", "server": "wiki-mcp",
             "tools": ["article.search", "article.read"], "required": true, "enforced": false } ]
  ```

- **D-7 — версии системных зависимостей.** v0.9 проверяет только наличие системной команды в `PATH` через `shutil.which`. Будущие требования к версиям могут объявляться только через встроенные probe-механизмы CocoaSkills. Манифесты не должны задавать исполняемые команды проверки версий, аргументы, shell-фрагменты, install-хуки или post-install-хуки.

## 24. План реализации v0.9

Минимальный послойный срез, каждый шаг можно тестировать независимо:

1. **Идентичность источника + допуск.** Нормализатор `csk.source_identity` — host/path, эквивалентность ssh/https/scp; сопоставление glob-шаблонов (`*`, `**`); `source_policy.authorization` с allow/deny/revoke.
2. **Только разбор манифестов.** `Skillfile.json` v2 (`sources`, `skills`, `includes`, `overrides`); `csk-skill.json` v4 (`dependencies.commands` / `.skills` / `.mcp_servers`). Диапазоны отклоняются.
3. **Модель данных разрешения.** `Node`, `Edge`, `Constraint`, `ActivationEdge`, `Closure`, `ResolvedNode`, `ResolvedInclude`.
4. **Построитель замыкания только для точных ссылок.** Разворачивание включений, поиск источников в корне, разворачивание зависимостей, обнаружение циклов, сведение ref→commit для одного имени навыка и обнаружение конфликтов.
5. **Планировщик активации.** Прямая / среда выполнения / контекстная активация; коллизии команд только по активным оберткам; статические проверки контракта по каждому ребру.
6. **Чтение/запись/status файла блокировки.** `Skillfile.lock`; обнаружение устаревания через `resolution_input_hash`; повторная проверка текущей политики при установке.
7. **Установка из файла блокировки.** Переиспользовать текущие примитивы контекста подсказки и среды выполнения; `--locked` никогда не выполняет повторное разрешение.
8. **Интеграция audit/status.** Замыкание, активные поверхности, эффективные возможности, метаданные MCP T1, совпадения политики источников только как диагностика.

## 25. Приемочные тесты v0.9

- `^1.0.0` в зависимости v0.9 → ошибка разбора.
- Две точные ссылки на один и тот же навык, разрешающиеся в **один commit**, → разрешено.
- Две точные ссылки, разрешающиеся в **разные commit**, → конфликт.
- Транзитивная зависимость без корневой записи `sources` → механизм разрешения падает.
- Источник включенного Skillfile допущен политикой и записан в файл блокировки.
- Собственные `sources` включенного Skillfile по умолчанию игнорируются.
- Отозванный источник проваливает установку даже со свежим файлом блокировки.
- Экспортируемая, но не активированная команда **не** конфликтует.
- Две активные обертки с одним именем команды конфликтуют.
- Команда навыка только в среде выполнения с `context.mode=none` и без потребительского контракта → предупреждение в обычном режиме, ошибка строгого аудита.
- `refs` потребительского контракта должны существовать и находиться под корнями включения контекста подсказки потребителя.
- `skill-wiki-memory` с `capabilities.exec: "none"` и `dependencies.commands.wk` → ошибка строгого режима.
- Глобально установленный поставщик **не** используется для удовлетворения замыкания проекта.
- Файл блокировки устаревает, когда меняется корневой или включенный Skillfile; **не обязательно устаревает**, когда меняется только политика, потому что политика не входит в `resolution_input_hash`; установка заново проверяет политику.
- Ребро `dependencies.commands.*.type=skill`, у которого поставщик не экспортирует эту команду с `type: script`, → ошибка разрешения.
- `branch` ref на ребре зависимости → ошибка разбора; `branch` у прямого/устаревшего навыка разрешается в конкретный commit, а `--locked` устанавливает этот commit даже после сдвига головы ветки.
- Источник допускается **до** любого clone/fetch; недопущенный источник никогда не скачивается.
- Существующий устаревший файл блокировки проваливает обычный `csk install`, не только `--locked`; отсутствующий файл блокировки создается при первом запуске.
- Системная зависимость `type: system` в v0.9 с заданными в манифесте `check`, аргументами команд, `script`, `install`, `post_install`, `version` или `version_probe` → ошибка разбора.
