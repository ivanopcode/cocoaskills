# CocoaSkills RFC 0001 — Current-Project Install and Project Init

Status: accepted
Date: 2026-05-15
Author: Ivan Oparin
Target: v0.2.x transition, v0.3.0 behavior flip

## 1. Goals

- Restore parity with conventional package manager UX: `cd project && csk install`.
- Separate project setup (one-time) from project operation (every run).
- Remove implicit global state mutation from the install flow.
- Make multi-project sync an explicit opt-in feature via `--all`.
- Preserve compatibility escape hatches for existing users with a one-cycle
  deprecation window.

## 2. Non-Goals

- Schema bump. Global config and `Skillfile.json` remain at `schema_version: 1`.
- Renaming the `projects` map to `checkouts`. Deferred to v0.4+ RFC.
- Registry sources, code signing, audit. Deferred to v0.4+.
- Removing the `projects` registry. It stays as opt-in convenience for `--all`.

## 3. Motivation

After v0.2.x, the install flow drifted from common package manager convention:

- `csk install` without arguments installs into every configured project at
  once. This is fleet-manager behavior, not package-manager behavior.
- `csk install .` auto-registers the current checkout into
  `~/.cocoaskills/config.json` as a side effect. The config grows over time
  even when the user never asked for it.
- `--fix-gitignore` on every install treats gitignore setup as recurring work
  when it is one-time per project.
- The global `projects` registry is required for any cross-project workflow
  but adds mental overhead for the single-project case.

CocoaPods, npm, cargo, poetry, and bundler all share the same pattern:

```
cd project
<tool> init     # once
<tool> install  # many times
```

v0.3.0 aligns `csk` with that pattern. Multi-project sync remains available
but explicit.

## 4. CLI Semantics Delta

| Command | v0.2.x | v0.3.0 |
|---|---|---|
| `csk init [path]` | n/a | NEW: create `Skillfile.json` + gitignore block |
| `csk install` | install all configured projects | install current project via walk-up |
| `csk install .` | install + auto-register | install, no config write |
| `csk install <alias>` | install configured project | install registered alias if present |
| `csk install /path` | install + auto-register | install, no config write |
| `csk install --all` | n/a | NEW: install every project in `projects` |
| `csk install --fix-gitignore` | append gitignore in-flight | deprecated escape hatch with warning |
| `csk status` | all configured projects | current project via walk-up |
| `csk status --all` | n/a | NEW: all registered projects |
| `csk upgrade` | update then install all | update then install current project |
| `csk upgrade --all` | n/a | NEW: update then install all registered |
| `csk bootstrap` | machine + project loop | machine config only |
| `csk project add` | required for sync | opt-in registration for `--all` |
| `csk project resolve` | resolve target | unchanged |
| `csk list` | configured projects | registered projects, for `--all` users |

## 5. `csk init` Specification

Single-purpose command. Creates project artifacts once.

### Synopsis

```
csk init [path] [--alias <name>] [--agents <list>] [--no-interactive]
```

### Behavior

Resolves target path to absolute. Defaults to CWD if omitted.

1. Resolve the target path to an absolute path and expand `~`.
2. If the target path does not exist, fail with
   `target path does not exist: <path>`. `csk init` does not create project
   directories; the user should `mkdir` or clone first.
3. Walk up from the target parent directory. If a `Skillfile.json` exists at
   any parent directory, fail with
   `already inside project at <parent>; nested projects unsupported`.
4. If target has `Skillfile.json` already, leave it untouched.
5. Otherwise create `Skillfile.json` with:
   ```json
   {
     "schema_version": 1,
     "project": { "alias": "<derived or --alias>" },
     "agents": ["<from --agents or global default_agents>"],
     "skills": []
   }
   ```
6. Add the managed `.gitignore` block:
   - If `.gitignore` exists, append the managed block when entries are missing.
   - If `.gitignore` does not exist, create it with the managed block.
   - This also happens outside git repositories; the user is explicitly
     initializing project-local generated state and may run `git init` later.
7. Do not create `.agents/`, `.claude/skills/`, or other adapter directories.
   That is `install`'s responsibility.
8. Do not fetch or install skills.
9. Do not register the project in global config.

### Defaults

- `--alias`: `basename(target)` sanitized via the same `_clean_alias` rules as
  `project_resolver`.
- `--agents`: global `default_agents` from `~/.cocoaskills/config.json`.
  If global config is absent or has no explicit default, use the implementation
  default `["codex_cli"]`. `csk init` should not require `csk bootstrap` first.

### Idempotency

Repeated invocations produce no diff. `Skillfile.json` is left as-is when
present. Managed gitignore block is added only when missing.

### Managed block format

```gitignore
# CocoaSkill
.agents/
.claude/skills/
.codex/skills/
.cursor/rules/
.gemini/skills/
```

Same block as today's `--fix-gitignore`. No duplicates on re-run.

### Errors

| Condition | Exit | Message |
|---|---|---|
| Walk-up finds parent `Skillfile.json` | 2 | `already inside project at <parent>; nested projects unsupported` |
| `--alias` cleaned to empty string | 2 | `--alias must contain at least one alphanumeric character` |
| Target path does not exist | 2 | `target path does not exist: <path>` |

## 6. `csk install` New Behavior

Without arguments, installs the current project resolved via walk-up.

Walk-up from CWD to filesystem root looking for `Skillfile.json`. If found,
install for that project. Resolved checkout alias is derived from the project
root and current git state via existing `project_resolver.resolve`, but **no
config write occurs**.

If walk-up finds no `Skillfile.json`:

```
error: no Skillfile.json found at or above <cwd>
  hint: run 'csk init' to create one
  hint: or run 'csk install /abs/path/to/project'
  hint: or run 'csk install --all' for multi-project sync (N projects configured)
```

The third hint appears only when the `projects` map is non-empty.

### Auto-Register Removed

`_cfg_and_alias_for_target` no longer writes the resolved checkout into
`~/.cocoaskills/config.json`. The in-memory `cfg` still carries the entry for
installer use during the run, but `save_config` is never called from this path.

`csk project add` remains the only way to mutate `projects`.

### Explicit Path and Tilde

```
csk install /Users/iv/Developer/...
csk install ./subdir
csk install ~/path/to/project
```

Walk-up applies if the path is a subdirectory of a project. No config write.
Tilde expansion via `Path.expanduser()` (existing behavior).

### Registered Alias

```
csk install partners-ios
```

Looks up `projects["partners-ios"]` in global config. Installs into that path.
Works only when the alias is already registered. No auto-create.

## 7. `csk install --all` Specification

Iterates `projects` from global config and installs each, in declaration
order. Equivalent of v0.2.x's bare `csk install`.

No filesystem discovery is performed. `--all` never scans `~/Developer`,
parent directories, or git worktrees for `Skillfile.json`. The only source of
truth for `--all` is the explicit `projects` map managed by
`csk project add`.

```
csk install --all
csk install --all --dry-run
csk install --all --strict-tags
```

`--all` is incompatible with a positional target. The CLI rejects
`csk install --all .` with:

```
error: --all cannot be combined with a target
```

Exit code: `0` if all projects succeeded, `1` if any project failed.

If `projects` is empty:

```
error: no registered projects; use 'csk project add' or run 'csk install' inside a project
```

## 8. `csk status` Semantics

- No arguments: status for current project via walk-up.
- With target: status for given path or alias.
- With `--all`: status for every entry in `projects`. Equivalent of v0.2.x's
  bare `csk status`.

Walk-up failure produces the same hints as `csk install`.

## 9. `csk upgrade` Semantics

Mirrors `csk install` exactly: defaults to current project, accepts target,
accepts `--all`. The fetch phase always covers all of `skills_root` regardless
of install scope.

## 10. `csk bootstrap` Change

Drops the interactive project loop. Asks only for machine settings:

1. `skills_root`
2. `preferred_locale`
3. `default_agents`
4. Shell hook prompt

Bootstrap closing message:

```
Wrote ~/.cocoaskills/config.json
Next steps:
  cd into a project directory
  csk init      # one-time per project
  csk install   # any time
```

If the user wants to pre-register projects for `--all`, they run
`csk project add` afterward.

## 11. Skillfile.json — No Schema Change

Schema remains at `schema_version: 1`. `project.alias` was added in v0.2.0 and
stays optional. No new fields.

## 12. Global Config — No Schema Change

Schema remains at `schema_version: 1`. `projects` stays in place. The only
behavioral change is that `csk` no longer writes new entries from the install
flow. Existing entries from v0.2.x deployments are preserved and used by
`csk install --all`.

## 13. Deprecation Warnings in v0.2.x

The final v0.2.x maintenance release emits warnings ahead of breaking changes.
Warnings go to stderr, do not change exit code, and fire at most once per
invocation per category.

Warning text is part of the public CLI contract for the transition window.
Each warning below must have an automated CLI test that asserts the relevant
command, exit code, and a stable identifying substring.

### Bare `csk install`

Triggered when invoked without a target while v0.2.x still uses legacy
all-project behavior. It fires whether the current directory is inside or
outside a CocoaSkill project, because v0.3.0 changes both cases.

```
csk install: WARNING — semantics will change in v0.3.0
  v0.3.0: 'csk install' without arguments installs only the current project
  v0.3.0: for multi-project sync, use 'csk install --all'
  This run uses legacy behavior over N configured projects.
```

### Bare `csk status`

Triggered when invoked without a target while v0.2.x still shows all configured
projects.

```
csk status: WARNING — semantics will change in v0.3.0
  v0.3.0: 'csk status' without arguments shows only the current project
  v0.3.0: for multi-project status, use 'csk status --all'
  This run uses legacy behavior over N configured projects.
```

### Bare `csk upgrade`

Triggered when invoked without a target while v0.2.x still upgrades all
configured projects.

```
csk upgrade: WARNING — semantics will change in v0.3.0
  v0.3.0: 'csk upgrade' without arguments updates skills and installs only the current project
  v0.3.0: for multi-project sync, use 'csk upgrade --all'
  This run uses legacy behavior over N configured projects.
```

### `csk install .` auto-register

Triggered on first auto-register of a new path within a session.

```
csk install .: WARNING — auto-register will be removed in v0.3.0
  v0.3.0: this command no longer modifies ~/.cocoaskills/config.json
  To pre-register this checkout for --all, run: csk project add <alias> <path>
```

### `--fix-gitignore` deprecation

Triggered whenever `--fix-gitignore` is passed.

```
--fix-gitignore: WARNING — deprecated for regular install flows
  Use 'csk init' once per project to set up gitignore.
```

## 14. Breaking Changes in v0.3.0

1. Bare `csk install` no longer installs all configured projects. Requires
   `--all`.
2. Bare `csk status` no longer reports on all configured projects. Requires
   `--all`.
3. Bare `csk upgrade` no longer upgrades all configured projects. Requires
   `--all`.
4. `csk install .` and `csk install /path` no longer mutate global config.
5. `csk bootstrap` no longer asks about projects.
6. `--fix-gitignore` remains functional but emits warning.

Migration cost for the typical user: read release notes once, optionally run
`csk init` in existing projects to drop `--fix-gitignore` from wrappers.

## 15. Error Message Standards

All new errors follow the shape:

```
error: <one-line problem>
  hint: <actionable first step>
  hint: <actionable alternative if applicable>
```

Exit codes unchanged: `0` success, `1` partial failure, `2` configuration or
usage error, `3` lock contention.

## 16. Test Surface Delta

Add:

- `csk init` creates `Skillfile.json` with derived alias
- `csk init` adds managed gitignore block when missing
- `csk init` is idempotent across consecutive invocations
- `csk init --alias` overrides derived name
- `csk init --agents` overrides default agents list
- `csk init` detects parent `Skillfile.json` and refuses
- `csk init` creates or updates `.gitignore` even outside a git repository
- `csk install` without args walks up to find `Skillfile.json`
- `csk install` without args errors cleanly when none found
- `csk install` without args does not write global config
- `csk install --all` iterates registered projects
- `csk install --all` rejects positional target
- `csk install --all` errors cleanly on empty `projects`
- `csk status` without args reports current project only
- `csk status --all` reports all registered
- v0.2.x transitional warning for bare `csk install`
- v0.2.x transitional warning for bare `csk status`
- v0.2.x transitional warning for bare `csk upgrade`
- v0.2.x transitional warning for `csk install .` auto-register
- `csk install --fix-gitignore` emits deprecation warning
- `csk bootstrap` does not prompt for projects

- Tests that assert auto-register on `csk install .`
- Tests that assume bare `csk install` covers all configured projects

## 18. Out of Scope for v0.3.0

Deferred to v0.4+ RFC:

- `csk checkout list / gc / forget` commands
- Schema bump and `checkouts` data model
- Registry sources for skills
- Code signing and trust providers
- `csk audit` command

## 19. Release Sequencing

1. Final v0.2.x patch with `csk init`, `--all`, and deprecation warnings, but
   no no-argument behavior flip.
2. Two-week minimum window before v0.3.0 for public users. This can be shortened
   only for internal-only testing before announcement.
3. v0.3.0 release with breaking changes and `csk init`.
4. `make_agents.py` wrapper in `partners-app` switches to
   `csk install . --fix-gitignore` flow on v0.2.x patch, then to plain
   `csk install .` after the project gains `csk init` setup.

## 20. Implementation Breakdown

### Phase A — v0.2.x maintenance patch

- Add `csk init`.
- Add `--all` to `install`, `status`, and `upgrade`.
- Add deprecation warnings from §13.
- Keep existing no-argument behavior unchanged.
- Add tests for every warning and for `csk init` idempotency.

This phase is safe to ship before the behavior flip.

### Phase B — v0.3.0 behavior branch

- Flip no-argument `install`, `status`, and `upgrade` to current-project
  resolution.
- Remove config writes from path-based install/upgrade flows.
- Keep explicit alias and `--all` flows working.
- Update tests that previously expected fleet-wide no-argument behavior.

### Phase C — documentation and release notes

- Update README happy path to `csk bootstrap`, `csk init`, `csk install`.
- Update CHANGELOG breaking-change section from §14.
- Update manual integration test plan sections listed in §17.

### Phase D — partners-ios migration

- Keep `make agents` as the explicit entry point.
- Remove wrapper-side global config mutation.
- Switch wrapper to `csk install .` once project setup is committed.
