# CocoaSkills RFC 0004 — Global Skills

Status: accepted
Date: 2026-05-27
Author: Ivan Oparin
Target: v0.6.0

## 1. Goals

- Add a global skill scope that is available outside any project checkout.
- Let users install personal or organization-wide baseline skills once and use
  them from all supported agent systems.
- Keep global skills independent from project-local skills. A project can pin a
  different version of the same skill without corrupting the global install.
- Reuse existing `Skillfile.json` schema v1 for global declarations.
- Reuse the shared runtime store:
  `~/.cocoaskills/runtime/<skill>/<commit>/`.
- Preserve user-owned content in user-level agent directories.
- Extend `csk shell-init` so global command shims are available everywhere,
  while project-local shims shadow global shims inside project checkouts.
- Add one-shot update and upgrade commands for all global skills.

## 2. Non-Goals

- Replacing project-local skill declarations. Projects still declare their own
  skills in their own `Skillfile.json`.
- Using global skills as implicit project dependencies.
- Best-version or cross-scope dependency resolution. Each scope pins its own
  refs independently.
- Public registry, discovery service, signing, audit, or trust providers.
- Installing system dependencies. `csk` continues to validate `type: system`
  dependencies but never installs them.
- Changing `Skillfile.json` schema. Global scope uses schema version `1`.
- Removing `csk install --all` or the existing `projects` map.
- Requiring global skills for normal project operation.

## 3. Motivation

Project-local CocoaSkills works for repository-specific agent context, but
some skills are not tied to a single repository:

- Personal productivity skills used in arbitrary directories.
- Organization-wide operational skills such as GitLab, Grafana, or incident
  helpers.
- Skills needed when a user starts Codex, Claude Code, Cursor, or Gemini from
  `~`, `/tmp`, or a scratch directory with no project `Skillfile.json`.

Today every project must duplicate those skills in its own `Skillfile.json`, or
the user loses access outside a managed checkout. This is unnecessary because
several csk primitives are already global:

- source repositories under `skills_root`;
- the snapshot cache;
- the runtime store keyed by `<skill>/<commit>`;
- the process lock under `~/.cocoaskills/.lock`;
- adapter ownership tracking through `.csk-managed.json`.

v0.6.0 adds a first-class global scope on top of those primitives.

## 4. Existing Primitives Reused

| Component | Current behavior | Global-scope readiness |
|---|---|---|
| `~/.cocoaskills/runtime/<skill>/<commit>/` | Shared runtime store | Reused unchanged |
| `skills_root` | Shared source repositories | Reused unchanged |
| Snapshot cache | Shared commit snapshots | Reused unchanged |
| Global lock | Serializes installs and cache/runtime writes | Reused unchanged |
| `.csk-managed.json` | Tracks csk-owned adapter entries | Reused for user-level adapters |
| `Skillfile.json` schema v1 | Project manifest | Reused for global manifest |
| Runtime GC | Scans project markers only | Extended to scan global markers |

## 5. Global Layout

`csk global init` creates the global scope under the existing csk home:

```
~/.cocoaskills/
├── config.json
├── global/
│   ├── Skillfile.json
│   ├── skills/
│   │   └── <skill>/
│   │       ├── SKILL.md
│   │       ├── references/
│   │       ├── agents/
│   │       └── .csk-install.json
│   ├── bin/
│   │   └── <command>
│   ├── env.sh
│   └── env.ps1
├── cache/
├── runtime/
└── .lock
```

Canonical global skill content lives at:

```
~/.cocoaskills/global/skills/<skill>/
```

Global command shims live at:

```
~/.cocoaskills/global/bin/
```

The runtime store remains unchanged:

```
~/.cocoaskills/runtime/<skill>/<commit>/
```

## 6. Global Skillfile

Global skills are declared in:

```
~/.cocoaskills/global/Skillfile.json
```

The file uses the existing schema version `1`:

```json
{
  "schema_version": 1,
  "agents": ["codex_cli", "claude_code", "cursor", "gemini"],
  "locale": "ru",
  "skills": [
    {
      "name": "skill-metrics",
      "git": "git@gitlab.example.com:skills/skill-metrics.git",
      "tag": "v1.0.0"
    },
    {
      "name": "skill-tracker",
      "git": "git@gitlab.example.com:skills/skill-tracker.git",
      "tag": "v1.1.0"
    }
  ]
}
```

`project.alias` is optional and SHOULD be omitted in global manifests. If it is
present, it is ignored for global-install identity. The global install target is
always named `global`.

`global` is an install-target identity used in CLI output and internal result
objects. It does not require a new marker schema field. Markers remain ordinary
`.csk-install.json` files stored under
`~/.cocoaskills/global/skills/<skill>/`.

`agents`, `locale`, `skills`, `source`, `git`, `tag`, `branch`, and `revision`
keep their existing meaning.

## 7. CLI Surface

Global operations live under an explicit `global` namespace:

```
csk global init
csk global add <name> [--git <url>] (--tag <tag> | --branch <branch> | --revision <sha>) [--source <source>]
csk global remove <name>
csk global list
csk global status
csk global install
csk global update
csk global upgrade
```

This is intentionally distinct from:

- `csk install`: current project;
- `csk install --all`: registered projects from the `projects` map;
- `csk global install`: user-wide global scope.

The namespace avoids an overloaded `--global` flag and keeps the three scopes
visibly separate.

## 8. Command Semantics

### `csk global init`

Creates `~/.cocoaskills/global/Skillfile.json` when missing:

```json
{
  "schema_version": 1,
  "agents": ["codex_cli"],
  "skills": []
}
```

Default agents come from global config `default_agents`. If global config is
missing or does not define defaults, `csk global init` falls back to
`["codex_cli"]`, matching `csk init` behavior.

Also creates:

- `~/.cocoaskills/global/skills/`
- `~/.cocoaskills/global/bin/`
- `~/.cocoaskills/global/env.sh`
- `~/.cocoaskills/global/env.ps1`

The command is idempotent. If the Skillfile already exists, it is left
untouched.

### `csk global add`

Convenience manifest editor. It updates the global Skillfile only; it does not
install skills.

Rules:

- The global Skillfile is created first if missing.
- Exactly one of `--tag`, `--branch`, or `--revision` is required.
- `--git` is optional but recommended for self-contained setup.
- `--source` is optional and follows normal Skillfile semantics.
- Adding an existing skill name replaces that declaration after validation.

The Skillfile remains the source of truth. Users may edit it directly.

### `csk global remove`

Removes a skill declaration from the global Skillfile. It does not immediately
delete installed files. The next `csk global install` performs cleanup.

### `csk global list`

Prints declared global skills and refs from the global Skillfile.

### `csk global install`

Installs all declared global skills using current local source refs. It does
not fetch.

Behavior mirrors project install:

1. Load global Skillfile.
2. Resolve or clone missing source repos through the existing `git` field
   behavior.
3. Resolve refs.
4. Build snapshots.
5. Validate `csk-skill.json` and system dependencies.
6. Install runtime roots or single-file commands into the shared runtime store.
7. Install canonical global context under `~/.cocoaskills/global/skills/`.
8. Install global shims under `~/.cocoaskills/global/bin/`.
9. Refresh user-level adapters.
10. Remove global skills that were previously csk-managed but are no longer
    declared.

If one skill fails, the global install result is non-zero and reports that
skill. Skills that can be installed are still installed.

### `csk global update`

Fetches source repositories for globally declared skills. It does not modify
global installed context, adapter dirs, or shims.

If a declared source repo is missing and the Skillfile has a `git` field,
`global update` MAY clone it. This matches user expectation that update
prepares all declared sources. It still does not install into adapters.

### `csk global upgrade`

Runs `csk global update`, then `csk global install`.

It does not touch project-local installs.

### `csk global install --dry-run`

Dry-run mirrors project install dry-run:

- validates the global Skillfile;
- clones missing source repos only into ephemeral temp directories;
- resolves refs;
- validates snapshots, skill specs, command collisions, and system
  dependencies;
- reports what would happen;
- writes nothing to `skills_root`, `~/.cocoaskills/global/`, adapter dirs,
  runtime store, cache, or shell env files.

### `csk global status`

Shows declared global skills versus installed state:

```
Global skills (~/.cocoaskills/global)
  skill-metrics       tag      v1.0.0       6234e2a  up-to-date
  skill-tracker      tag      v1.1.0       548205e  update-available
  skill-logbook       branch   main         -        missing
```

Status labels match project status where possible:

- `up-to-date`
- `missing`
- `update-available`
- `content-drift`
- `error`

## 9. User-Level Adapters

Global adapters are installed into user-level agent directories:

| Agent | Global adapter directory |
|---|---|
| Codex CLI | `~/.codex/skills` |
| Claude Code | `~/.claude/skills` |
| Cursor | `~/.cursor/rules` |
| Gemini | `~/.gemini/skills` |

Each adapter directory uses the same ownership model as project adapters:

```
~/.claude/skills/
├── .csk-managed.json
├── skill-metrics -> ~/.cocoaskills/global/skills/skill-metrics
└── handwritten-user-skill/
```

Rules:

- `csk` may create the adapter directory.
- `csk` may create, replace, or delete only entries listed in
  `.csk-managed.json`.
- Unmanaged entries are never deleted.
- If a target path exists and is not csk-managed, install fails with a clear
  unmanaged conflict error.
- Adapter mode follows global config `adapter_mode`: `auto`, `symlink`, or
  `copy`.

## 10. Agent Lookup Semantics

`csk` provides two physical scopes:

- project adapters under the project checkout;
- global adapters under user home.

The intended effective behavior is:

1. Inside a project, project-local skills shadow global skills.
2. Outside any project, global skills are available.

`csk` MUST NOT rely on undocumented agent behavior without smoke tests. For
each supported adapter, distribution/manual tests should verify that the agent
observes project-local skills before user-global skills when both exist.

If a specific agent does not support a user-level skill directory, that adapter
must document the limitation. Global install should still install the scopes
that are supported.

## 11. Global Command Shims

Global command shims are written to:

```
~/.cocoaskills/global/bin/
```

Unix:

- shims are symlinks to the runtime store;
- executable bits are enforced on runtime entrypoints as today.

Windows:

- shims are `.cmd` wrappers, same as project shims.

Within global scope, command collisions are errors. Two global skills cannot
export the same command.

Between global and project scopes, command name collisions are allowed. Project
scope shadows global scope through PATH ordering.

## 12. Shell Activation

`csk global install` writes:

```
~/.cocoaskills/global/env.sh
~/.cocoaskills/global/env.ps1
```

`env.sh` prepends:

```
~/.cocoaskills/global/bin
```

`csk shell-init` is extended to activate global commands by default.

Required PATH order inside a project:

```
<project>/.agents/bin
~/.cocoaskills/global/bin
<existing PATH>
```

Required PATH order outside any project:

```
~/.cocoaskills/global/bin
<existing PATH>
```

This means:

- a project-local `yt` shadows a global `yt` while the shell is inside that
  project;
- the global `yt` is still available everywhere else.

`csk shell-init <shell> --no-global` MAY be added in v0.6.0 as an escape hatch.
If omitted from v0.6.0, it remains a valid future addition.

## 13. Conflict Semantics

### Same skill name, different scopes

Allowed.

Example:

- Global Skillfile pins `skill-tracker` at `v1.1.0`.
- Project Skillfile pins `skill-tracker` at `v1.3.0`.

Results:

- Global canonical context:
  `~/.cocoaskills/global/skills/skill-tracker`
- Project canonical context:
  `<project>/.agents/skills/skill-tracker`
- Runtime store contains both commits if needed:
  `~/.cocoaskills/runtime/skill-tracker/<commit-v1.1.0>`
  and
  `~/.cocoaskills/runtime/skill-tracker/<commit-v1.3.0>`

No marker, adapter, or runtime path collides.

### Same command name, different scopes

Allowed.

PATH order determines runtime behavior:

- inside project: project command wins;
- outside project: global command wins.

### Same skill or command inside one global Skillfile

Rejected.

- Duplicate skill name: manifest validation error.
- Duplicate command export: install planning error.

## 14. System Dependencies and Security

Global skills follow the same security model as project skills:

- `type: system` dependencies are checked with `shutil.which`.
- Missing system dependencies block installation of that skill before writes.
- `csk` never installs system commands.
- `csk-skill.json` schema v2 continues to reject execution-like fields such
  as `install`, `check`, `post_install`, `script`, and `command_args`.

Global scope must not weaken manifest execution guarantees. Global install is
still a copy/symlink operation plus git operations, not arbitrary skill code
execution.

## 15. Runtime GC

Runtime GC currently scans project markers:

```
<project>/.agents/skills/*/.csk-install.json
```

v0.6.0 extends GC to also scan global markers:

```
~/.cocoaskills/global/skills/*/.csk-install.json
```

A runtime entry may be deleted only when no project marker and no global marker
references the same `<skill>/<commit>`.

This is required because a runtime may be used only by global skills.

## 16. Locking

The existing global lock remains the only lock:

```
~/.cocoaskills/.lock
```

`csk global install`, `csk global update`, and `csk global upgrade` acquire the
same lock as project installs. This intentionally serializes writes to:

- source repositories;
- snapshot cache;
- runtime store;
- global adapters;
- global bin.

No finer-grained lock is introduced in v0.6.0.

## 17. Backwards Compatibility

- Existing project commands are unchanged.
- Existing `~/.cocoaskills/config.json` remains schema version `1`.
- Existing project `Skillfile.json` remains schema version `1`.
- Users without `~/.cocoaskills/global/` see no behavior change until they run
  a `csk global` command.
- `csk install`, `csk install --all`, `csk status`, and `csk upgrade` retain
  their v0.5 behavior.
- Runtime store layout remains unchanged.

## 18. Error Message Standards

Errors should follow existing csk style: concise message plus actionable path
or hint.

Examples:

```
Global Skillfile not found: ~/.cocoaskills/global/Skillfile.json
  Run 'csk global init' first.
```

```
Global adapter target already exists and is not managed by csk:
  ~/.claude/skills/skill-metrics
```

```
Missing system command 'glab' for skill-review.
  Hint: Install the review CLI through project bootstrap tooling.
```

```
Command collision for 'yt': exported by skill-tracker and skill-logbook
```

## 19. Test Surface

Required tests:

- `csk global init` creates the global Skillfile, skills dir, bin dir, and env
  files.
- `csk global init` is idempotent.
- `csk global add` adds or replaces a valid declaration.
- `csk global remove` removes a declaration and the next install cleans it up.
- `csk global remove <missing>` fails cleanly with a configuration error and
  does not traceback.
- `csk global install --dry-run` validates inputs and writes nothing to
  `skills_root`, global scope, adapters, runtime store, cache, or env files.
- `csk global install` installs canonical context under
  `~/.cocoaskills/global/skills`.
- `csk global install` is idempotent: running it twice with unchanged
  Skillfile and source refs leaves already up-to-date skills untouched.
- `csk global install` creates user-level adapter entries and preserves
  unmanaged user content.
- User-level adapter conflict with unmanaged entry fails cleanly.
- Global command shims point to `~/.cocoaskills/runtime`.
- Project and global installs can pin the same skill at different commits.
- Project command shim shadows global command shim through shell-init PATH
  order.
- `csk global status` reports `missing`, `up-to-date`,
  `update-available`, and `content-drift`.
- `csk global update` fetches source repos without changing installed context.
- `csk global upgrade` updates then installs.
- Runtime GC preserves a runtime entry referenced only by a global marker.
- Missing `type: system` dependency blocks only that skill and does not write
  partial global context.
- Lock contention between `csk global install` and project install returns the
  existing lock exit code.
- Windows `.cmd` global shims are generated through the same abstraction as
  project shims.

Recommended integration/manual tests:

- Claude Code sees a global skill outside a project checkout.
- Codex CLI sees a global skill outside a project checkout.
- Cursor sees a global rule outside a project checkout.
- Gemini sees a global skill outside a project checkout.
- Inside a project, project-local version wins over global version for both
  agent context and command shims.

## 21. Release Sequencing

1. Implement RFC 0004 behind normal CLI commands. No deprecation window is
   required because the feature is additive.
2. Release `v0.6.0-rc1` to TestPyPI if the implementation touches shell-init
   or adapter code substantially.
3. Smoke global install on macOS and Linux.
4. Release `v0.6.0` to PyPI and GitHub Releases.
5. Bump Homebrew formula.
6. Run distribution smoke with Homebrew included.
7. Validate one real global skill in the demo-app environment:
   - install globally;
   - start shell outside any project;
   - confirm command is available;
   - start agent outside any project;
   - confirm skill is visible.
